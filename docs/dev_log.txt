Task:
Implemented a project-wide runtime logging system using Python's standard `logging` library.

What changed:
Added a shared logging configuration module in `logic/logging_config.py` and integrated it across the main Flask application, database helper functions, and the database setup workflow. The application now writes runtime logs to `logs/app.log` using a rotating file handler, while also sending log output to the console during development.

Why this was done:
The project needed a proper runtime logging system that was separate from the development log. The development log is for documenting implementation progress, technical reasoning, testing, and reflection. Runtime logging serves a different engineering purpose: it records application behaviour during execution so that requests, database access, setup operations, and future allocation workflows can be traced and debugged more effectively.

Design reasoning:
Python's built-in `logging` library was chosen because it is the standard and most widely used logging approach in Python applications. A shared configuration module was used so that logging behaviour is consistent across the project rather than being configured separately in multiple files. A rotating file handler was introduced to prevent the log file from growing indefinitely over time.

Files affected:
- `logic/logging_config.py`
- `app.py`
- `logic/database_helpers.py`
- `setup/setup_database.py`
- `README.md`
- `.gitignore`

Problem encountered:
The initial integration caused `python setup/setup_database.py` to fail because the shared logging configuration module could not be imported when the script was executed directly from the `setup` directory context. This happened because the project root was not automatically available on `sys.path` in that execution mode.

Solution applied:
Updated the setup script so that the project root is inserted into `sys.path` before importing the shared logging configuration. This restored compatibility with the documented setup command and allowed the logging system to be reused in the setup workflow without duplicating configuration logic.

Testing performed:
Ran the automated test suite using `./venv/bin/python -m pytest -q`, which passed successfully with 6 tests passing. Also ran `./venv/bin/python setup/setup_database.py` to confirm that the database setup still worked after the logging integration. Verified that `logs/app.log` was created and contained application and database log entries.

Changes to original plan:
An earlier attempt briefly created a separate development log file outside `docs`, but this was corrected so that the project development log remained in the `docs` folder while runtime execution logs were moved into the dedicated `logs` directory.

Reflection:
This change improved the project structure by separating two forms of logging that serve different purposes. The development log remains useful for academic reporting and design justification, while the runtime logging system now provides a maintainable technical foundation for debugging and tracing future features such as allocation logic, supply aggregation, and database-driven coordination workflows.
